# ポイッと（POITTO）— Claude Code への指示プロンプト

このファイルは、Claude Code にアプリ実装を依頼するための指示書です。下の「■ 貼り付けるプロンプト」をそのままコピーして Claude Code に貼ってください。

---

## 事前準備（1回だけ）

1. アプリ用の新しいフォルダ（リポジトリ）を作る。例：`poitto-app`
2. そこへ、これまでの資料と素材をコピーして入れる：
   - `証憑ポスト_MVP実装仕様書.md`（← 実装の詳細。最重要）
   - `証憑ポスト_要件定義書_AWS版.docx`（← インフラ/AWS構成）
   - `assets/` フォルダ一式（ロゴ・アイコン・favicon・README.md）
3. そのフォルダで Claude Code を開き、下のプロンプトを貼る。

> メモ：資料内の旧コードネームは「証憑ポスト」ですが、**プロダクト名は「ポイッと（英字 POITTO）」**です。UIの表示名・ロゴは「ポイッと / POITTO」を使ってください。

---

## ■ 貼り付けるプロンプト（ここからコピー）

```
あなたはこのリポジトリでWEBアプリ「ポイッと（POITTO）」を実装します。
電子帳簿保存法対応の証憑（請求書・領収書）自動ファイリングSaaSです。
証憑を投函するだけで、AIが項目抽出→統一名にリネーム→月別に自動保存→索引化します。

# 最初に必ず読むもの
1. `証憑ポスト_MVP実装仕様書.md` … データモデル・AI抽出契約・API・画面・受入基準。実装の正。
2. `証憑ポスト_要件定義書_AWS版.docx` … インフラ（Vercel＋AWS）と非機能要件。
3. `assets/README.md` … ブランド素材と配色。
読んだら、理解した要点と作業計画（フェーズ分け）を先に箇条書きで提示してください。

# 技術スタック（確定）
- フロント/UI: Next.js(App Router, TypeScript) + Tailwind CSS。レスポンシブ＋PWA前提。デプロイはVercel。
- バックエンド: AWS（東京リージョン）。認証=Cognito、API=API Gateway+Lambda、
  DB=Aurora Serverless v2(PostgreSQL)、保管=S3、非同期抽出=S3→SQS→Lambda、秘密情報=Secrets Manager、IaC=AWS CDK。
- AI抽出: Google Gemini Flash系。既定はFlash-Lite、低確信のみ上位Flash。JSON Schema構造化出力。

# 設計原則（厳守）
- 移植性: DBアクセス/ストレージ/認証/AI抽出は必ずアダプタ(インターフェース)越しに使い、ベンダー固有APIをアプリ全域に散らさない。将来AWS→他クラウドやGemini→他モデルへ差し替え可能に保つ。
- SQLは標準PostgreSQLの範囲で書く。
- 秘密情報はコードに書かず環境変数/Secrets Manager。
- UIコピーは日本語、やさしい言い回し。配色は下記ブランドカラー。

# ブランド
- 表示名「ポイッと」/ 英字「POITTO」。ひとこと「入れるだけで、証憑がかたづく。」
- カラー: コーラル #E8542B(主色) / ミント #1D9E75(完了・OK) / アンバー #EF9F27(要確認) / インク #2C2C2A(文字) / 背景 白〜オフホワイト。
- 素材(assets/)をそのまま使う: ヘッダー=logo/poitto_logo_horizontal.png、favicon=favicon/favicon.ico、
  PWAアイコン=app-icon/poitto_appicon_512.png、空状態やロード=mark/poitto_mark_256.png。
  まず `public/brand/` にコピーして参照すること。
- 命名ルール: 取引年月日_取引先名_取引金額_書類の種類（例 260630_佐川急便株式会社_71610_請求書.pdf）。金額は税込。

# 進め方（重要：フェーズごとに区切り、各フェーズ完了時に必ず一旦止まって報告し、私の承認を待つこと。勝手に全部進めない）

## フェーズ1: 土台とブランド
Next.js(TS)+Tailwind の雛形。ブランドカラーをTailwindテーマに定義。素材をpublic/brandへ配置。
サイドバー＋ヘッダーの共通レイアウト（ホーム/投函/月別一覧/検索/設定）。レスポンシブ。favicon設定。
→ モックデータのみ。動く画面骨格を見せて一旦停止。

## フェーズ2: 画面（モックデータ）
実装仕様書§7の画面を作る: ダッシュボード(サマリーカード)、投函ボックス(ドロップゾーン)、
確認キュー(原本プレビュー横に抽出項目フォーム/低確信ハイライト)、月別ビュー(月セレクタ＋月別の件数・合計・一覧)、
証憑一覧・検索(日付/金額/取引先、範囲・複合条件)。すべてモックデータで見た目を完成。→ 停止。

## フェーズ3: 抽出アダプタ＋業務フロー（モック抽出）
`Extractor` インターフェース(実装仕様書§5の型)を定義し、まずモック実装で
アップロード→抽出→命名→月別保存→一覧、を通す。保存先やDBは当面ローカル/インメモリで可。→ 停止。

## フェーズ4: Gemini抽出
`GeminiExtractor` を実装(§5のプロンプト・JSON Schema、env GEMINI_API_KEY)。
テキストPDFはネイティブ抽出、確信度で要確認振り分け。→ 停止。

## フェーズ5: AWSバックエンド
CDKでCognito/API Gateway/Lambda/Aurora(PostgreSQL,§3のDDL/RLS)/S3/SQS/Secrets を定義。
DB/ストレージ/認証をアダプタ実装で接続。フロントはCognito JWTでAPIを呼ぶ。→ 停止。

## フェーズ6: 仕上げ
PWA化(manifest/service worker、スマホのカメラ投函)。レスポンシブ調整。
受入テスト(下記)を満たすことを確認。→ 停止。

# 受け入れ基準（サンプルPDFで検証）
サンプル請求書(佐川急便/2026-06-30締/税込71,610円)を投函したとき:
- 抽出: 日付=2026-06-30, 取引先=佐川急便株式会社, 税込金額=71610, 種別=請求書, 登録番号=T8130001000053
- ファイル名=260630_佐川急便株式会社_71610_請求書.pdf、保存先=保存済み/2026年06月/
- 一覧に表示、取引先「佐川」で検索ヒット、6/1〜6/30の範囲でヒット、CSV出力に1行
- 監査ログに create/extract/confirm、別テナントからは不可視、スマホ幅(375px)で崩れない

まずフェーズ1から。着手前に作業計画を提示してください。
```

## 使い方のコツ
- 各フェーズの終わりで止まるので、動作を確認してから「次のフェーズへ」と伝える。
- 途中で仕様を変えたくなったら、実装仕様書 or この指示のフェーズ記述を書き換えてから再開すると、齟齬が出ない。
- AWS部分(フェーズ5)は費用と権限が絡むので、AWSアカウントとIAM権限を用意してから着手するとスムーズ。
- まず動くものを早く見たい場合は「フェーズ4まではローカルで完結、AWSは後で」と伝えてもOK。
